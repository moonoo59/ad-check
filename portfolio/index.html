문제와 해결

Before / After 비교로 본 프로젝트의 존재 이유

Before — 수작업 프로세스
  • ▶ 요청자(채널 담당자)가 이메일로 채널·방송일시·광고주를 담당자에게 요청
  • ▶ 담당자(대표 담당자)가 Logger Storage(NAS)를 직접 탐색하며 파일을 수동 식별
  • ▶ 자정 넘김·날짜 오산으로 인한 오파일 전달 사고
  • ▶ 요청 내역·처리 결과 추적 불가 (엑셀 수기 관리)
  • ▶ NAS 마운트 장애 시 복사 작업 전체 중단
After — 웹 자동화 시스템
  • ▶ 요청자(채널 담당자)가 웹에서 직접 요청 등록 → 실시간 상태 조회
  • ▶ 파일명 매칭 알고리즘이 자동으로 후보 파일 탐색 및 점수 산출
  • ▶ 자정 넘김·KST/UTC 처리를 코드 레벨에서 완전 정규화
  • ▶ 모든 요청·승인·복사 이력을 감사 로그로 영구 보관
  • ▶ NAS 의존 제거 → 로컬 스토리지 복사 + 웹 다운로드 제공

기술 스택

선택 근거와 함께 분류별로 정리

Backend
Node.js Express TypeScript SQLite (better-sqlite3) express-session pnpm

사내망 단일 PC 환경에서 외부 DB 서버 없이 운영 가능한 SQLite 선택. Express는 미들웨어 체계가 간결하여 소규모 API 서버에 최적.

Frontend
React 18 TypeScript Vite React Router v6 Tailwind CSS Axios

Vite로 빌드 속도 확보, Tailwind로 디자인 시스템 없이도 일관된 UI 구현. React Router의 Outlet/RequireAuth 패턴으로 권한별 라우팅 분리.

Packaging / Infra
Electron electron-builder SMB (macOS Finder) macOS Keychain

Electron으로 Node 런타임·백엔드·프론트엔드를 단일 .app 번들로 패키징. SMB 자격증명은 macOS Keychain에만 저장, 코드에 일절 미포함.

개발 도구
ESLint pnpm workspaces concurrently ts-node-dev Claude Code

모노레포 구조로 백엔드·프론트엔드·Electron을 단일 pnpm workspace로 관리. pnpm dev로 백엔드(4000)·프론트(5173) 동시 기동.

핵심 기능

기능별 역할과 기술 포인트

요청 등록

채널·방송일시·요청 시간대·모니터링 송출 시각을 입력하면 요청이 생성되고, 백엔드가 즉시 Logger Storage 파일 탐색을 트리거합니다. 다수 광고를 한 요청 안에 복수 항목으로 묶어 일괄 처리 가능.

파일 탐색 & 검수

매칭 알고리즘이 파일별 점수(0~100)를 산출하고 match_reason을 한글로 설명. 담당자는 후보 목록에서 파일을 선택한 뒤 승인 또는 반려. 자정 넘김 시 다음날 디렉토리까지 자동 탐색.

파일 복사 & 다운로드

승인 후 Logger Storage → 서버 로컬 스토리지로 복사 진행. 완료 파일은 웹 다운로드 API로 직접 제공. 1일 경과 파일은 스케줄러가 자동 정리 (1시간 간격).

감사 로그

요청 생성·승인·반려·복사·삭제 등 모든 상태 변화를 audit_logs 테이블에 기록. 관리자 화면에서 날짜·사용자·액션 기준으로 검색 및 조회 가능.

통계 & CSV 내보내기

기간·채널·상태별 요청 현황 집계 대시보드. 요청 목록 전체를 CSV(Excel 호환) 형식으로 내보내기. KST 기준 날짜 필터 정확도 보장.

사용자 & 채널 관리

관리자가 사용자 계정·역할을 직접 관리. 채널 표시명 ↔ Logger Storage 폴더명 매핑을 UI에서 편집 가능. 채널 매핑 변경 이력도 별도 테이블로 보존.

기술적 하이라이트

설계 의사결정과 트러블슈팅 기록

문제

파일명 CH_20260305_235955_0100.avi에서 종료 시각(01:00)이 시작 시각(23:59)보다 작아 파일 범위가 음수로 계산됨.

원인

파일명의 마지막 HHMM은 종료 시각이지 길이가 아님. 단순 문자열 비교만으로는 자정 경계를 인식할 수 없음.

해결

endH < startH 조건 감지 시 종료 날짜에 +1일 보정. 동일 시 분 비교 케이스도 함께 처리.

backend/src/modules/files/file-matcher.ts — parseFileName 
// 종료 DateTime 구성 (처음엔 같은 날짜)
let endTime = new Date(year, month, day, endH, endM, 0);

// 자정 넘김 처리:
// - 종료 HH < 시작 HH: 명확한 자정 넘김
// - 종료 HH == 시작 HH && 종료 MM < 시작 MM: 동일 시 자정 넘김
if (
  endH < startH ||
  (endH === startH && endM < startM)
) {
  endTime = new Date(year, month, day + 1, endH, endM, 0);
}
매칭 점수 기준 (0~100점)
조건 점수
모니터링 송출시간이 파일 범위(start~end) 내 포함 +60
요청 시간대와 파일 범위가 겹침 (overlap 있음) +30
겹침 비율이 요청 범위의 50% 이상 +10

Path Traversal 방어

Logger Storage·로컬 전달 경로에 대한 모든 파일 접근을 path.resolve + prefix 검사로 루트 이탈 차단.

backend/src/common/path-guards.ts — traversal 차단 
function isWithinRoot(rootPath: string, targetPath: string): boolean {
  return targetPath === rootPath
    || targetPath.startsWith(`${rootPath}${path.sep}`);
}

export function resolvePathWithinRoot(
  rootPath: string, targetPath: string, label = '경로',
): string {
  const resolvedRoot   = path.resolve(rootPath);
  const resolvedTarget = path.resolve(targetPath);
  if (!isWithinRoot(resolvedRoot, resolvedTarget)) {
    throw new Error(`${label}가 허용된 루트 경로를 벗어났습니다.`);
  }
  return resolvedTarget;
}
문제

날짜 필터 적용 시 KST 00:00 이후 요청이 전날로 분류되는 현상. 최대 9시간 오차.

원인

초기 스키마 기본값 datetime('now', 'localtime')과 이후 new Date().toISOString()(UTC) 혼재.

해결

Migration 010으로 기존 KST 문자열을 일괄 UTC ISO 변환. 이후 common/datetime.ts로 KST 경계값 변환 일원화.

backend/src/config/migrations/010_normalize_timestamps_to_utc.sql — KST → UTC 변환 패턴 
-- 'T'가 없는 값 = KST 로컬 문자열 → UTC로 변환 (-9시간)
UPDATE users
SET
  created_at = CASE
    WHEN created_at IS NOT NULL AND instr(created_at, 'T') = 0
      THEN strftime('%Y-%m-%dT%H:%M:%fZ', datetime(created_at, '-9 hours'))
    ELSE created_at
  END,
  -- updated_at, reviewed_at 등 동일 패턴 반복
  ...;

-- 이후 신규 스키마 기본값은 UTC 강제:
created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now'))
이후 원칙: 날짜 필터·화면 표시 구현 시 직접 new Date() 비교 금지. 반드시 common/datetime.ts의 toKSTDateString() 유틸로 KST 경계값 변환 후 쿼리.

회고 & 학습

이 프로젝트를 통해 얻은 것과 다음 단계

배운 점
  • ▶ 도메인 지식이 설계를 결정한다. "종료 시각 = 길이가 아님"을 놓쳤으면 매칭 알고리즘 전체가 무너졌을 것.
