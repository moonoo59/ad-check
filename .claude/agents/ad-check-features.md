# 광고 증빙 요청 자동화 시스템 정리
## 1. 프로젝트 목적
광고팀이 이메일로 요청하는 방송 광고 증빙 업무를 웹 기반으로 단순화하여, 기술팀이 직접 스토리지에 접속해 파일을 찾고 복사해주는 반복 업무를 줄이는 것이 목적이다.
이 시스템의 핵심 목표는 다음이다.
 - 광고팀 요청을 표준화한다
 - 기술팀의 수작업 탐색/복사 시간을 줄인다
 - 요청/처리/복사 이력을 남긴다
 - 향후 광고 구간 추출, 시청률 연동, 대화형 검색 등으로 확장 가능하게 만든다
 - 비용은 최소화하고, 우선은 사용자 PC를 웹서버처럼 활용하는 구조로 시작한다
 - 이 시스템은 수익 창출용 서비스가 아니라 내부 업무 효율화 도구다.
## 2. 현재 업무(As-Is)
 - 현재 업무 흐름은 다음과 같다.
 광고팀이 이메일로 광고 증빙을 요청한다
기술팀이 Logger Storage에 저장된 송출 영상을 직접 찾는다
기술팀만 접근 가능한 방송망 NAS에서 필요한 시간대 파일을 수동 확인한다
해당 파일을 광고팀/기술팀 공유 NAS로 복사한다
광고팀이 공유 NAS에서 확인하거나 외부 전달에 활용한다
현재 문제점은 다음과 같다.
요청 양식이 메일 중심이라 표준화가 약하다
기술팀만 Logger Storage 접근 가능하여 병목이 발생한다
파일 탐색과 복사가 수작업이다
누가 언제 무엇을 요청했고 처리했는지 체계적으로 남지 않는다
향후 통계나 패턴 분석에 활용할 데이터가 축적되지 않는다
채널명 변경 이력이 있어 검색/매칭이 불편하다
3. 목표 업무(To-Be)
목표 흐름은 다음과 같다.
광고팀이 웹페이지에서 메일과 유사한 양식으로 요청한다
요청 항목을 기준으로 시스템이 자동으로 대상 파일을 탐색한다
기술팀이 확인 버튼을 눌러 최종 승인한다
시스템이 Logger Storage에서 대상 파일을 찾아 광고팀/기술팀 공유 NAS로 복사한다
처리 결과와 이력이 자동 저장된다
광고팀은 결과를 확인하고 필요한 경우 전달한다
즉, 기술팀은 직접 파일을 찾는 작업자에서 확인/승인 관리자로 역할이 바뀐다.

5. 핵심 요구사항
5-1. 요청 접수 기능
광고팀은 이메일 대신 웹페이지에서 요청을 등록해야 한다.
입력 항목은 최소 다음 수준이 필요하다.
요청자
요청일시
영업담당자
채널
광고주
방송일자
파일 요청 시간대
모니터링 송출 시간
비고/추가 요청사항
실제 메일 양식도 이 구조와 유사하므로, 기존 사용 습관을 크게 바꾸지 않는 방향이 적절하다. 
request_email
요청은 한 건에 여러 줄(여러 광고주/여러 시간대)을 넣을 수 있어야 한다.
5-2. 파일 탐색 기능
기술팀이 “확인” 버튼을 누르면 시스템이 자동으로 Logger Storage 내에서 대상 파일을 찾는다.
탐색 기준은 다음을 조합한다.
채널
방송일자
요청 시간대
모니터링 송출 시간
채널 매핑 정보
탐색 결과는 단순히 1건만 보여주는 것이 아니라, 기술팀이 검증할 수 있게 리스트업되어야 한다.
예시:
원본 파일 경로
파일명
시작/종료 시간
파일 크기
생성 시각
예상 매칭 근거
5-3. 복사 기능
확인된 대상 파일은 다음 경로로 복사된다.
출발: Logger Storage(송출영상 녹화)
도착: 광고팀/기술팀 공유 NAS
복사는 자동화하되, 초기에 반드시 기술팀 승인 후 실행되도록 하는 것이 안전하다.
복사 결과로 다음 상태가 필요하다.
대기
탐색 완료
승인 완료
복사 중
복사 완료
실패
반려
5-4. 채널명 매핑 기능
과거와 현재 채널명이 달라 매핑 기능이 반드시 필요하다.
예시:
ESPN → Sports
Golf → Golf
NICK → Golf2
Plus → Plus
ETV → funE
CNBC → Biz
Fil → Life
이 매핑은 운영 중 변경될 수 있으므로 하드코딩하면 안 된다.
관리자 화면에서 수정 가능해야 하며, 이력도 남기는 것이 좋다.
즉, 이 시스템은 표시용 채널명과 실제 저장 경로 기준 채널명을 분리해서 관리해야 한다.
5-5. 처리 이력/로그 기능
통계와 추후 분석을 위해 기록을 남겨야 한다.
최소 기록 항목은 다음이 필요하다.
누가 요청했는지
언제 요청했는지
어떤 광고주/채널/시간대를 요청했는지
누가 검토했는지
승인/반려 여부
복사 실행 시각
복사 성공/실패 여부
결과 파일 위치
오류 메시지
이 기록은 단순 감사 용도뿐 아니라, 나중에 자주 요청되는 채널/광고주/시간대 분석에도 활용 가능하다.
5-6. 보안 요구사항
이 요구사항은 매우 중요하다.
Logger Storage 접근은 항상 열려 있으면 안 되고,
사용자가 출근해서 서비스를 켤 때만 마운트되고, 끄면 언마운트되어야 한다.
즉 다음 조건이 필요하다.
서비스 시작 시 볼륨 mount
서비스 종료 시 볼륨 unmount
관리자 화면에서 수동 mount/unmount 가능
현재 마운트 상태 표시
mount/unmount 실행 이력 저장
비정상 종료 시 다음 기동 시 상태 점검
보안 관점에서 이 부분은 단순 편의 기능이 아니라 핵심 제어 기능이다.
즉, 시스템은 항상 열려 있는 상시 서버가 아니라
운영자가 통제하는 세션형 내부 서비스 성격이 강하다.
5-7. 운영 환경 요구사항
초기에는 비용을 최소화해야 하므로 다음 구조가 적절하다.
사용자의 PC가 웹서버 역할 수행
Logger Storage와 공유 NAS는 네트워크 마운트 방식
내부망 전용 사용
사용 시에만 서비스 실행
향후 정규화되면 별도 서버 구성까지 고려한다.
즉, 설계는 2단계가 필요하다.
1단계
내 PC 기반
저비용
수동 기동/종료
소수 사용자
2단계
별도 서버
상시 서비스
권한 체계 강화
통계/확장 기능 본격화

6-3. 추후 확장 기능
말씀하신 방향과 대화 내용을 종합하면, 확장 기능은 아래처럼 정리된다.
광고 구간만 추출
현재는 1시간 단위 .avi 파일을 복사하지만, 추후 특정 광고 송출 시점만 잘라서 추출하는 기능으로 확장 가능하다.
필요 조건:
모니터링 송출 시간 정확도 확보
구간 추출 기준 정의
ffmpeg 등 도구 활용 검토
시청률 데이터 연동
광고가 나간 시점의 시청률 환경을 조회하는 기능
주의점:
이건 “광고 효과” 단정이 아니라 “광고가 노출된 시점의 시청률 환경” 제공으로 표현해야 한다
보통 D+1 데이터 수신 구조를 가정해야 한다
대화형 검색
예:
“대웅제약 광고 요청 내역 보여줘”
“지난달 골프 채널 요청 건수 알려줘”
“가장 많이 요청된 광고주는 누구야”
초기에는 메뉴 기반으로 시작하고, 나중에 대화형 인터페이스를 붙이는 게 현실적이다.
