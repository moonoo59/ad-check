> 작성일: 2026-03-24
> 목적: 현재 문서와 작업 흐름이 섞여 보이는 문제를 줄이기 위해, 어떤 문서가 무엇의 기준인지와 지금 당장 해야 할 작업 범위를 정리한다.

## 1. 왜 이 문서가 필요한가

현재 저장소에는 아래 성격이 다른 문서가 함께 존재한다.

- 과거 분석 문서
- 완료된 개선사항 기록
- 현재 오픈된 감사 이슈
- 미래 고도화 설계
- 역사 누적형 진행 로그

이 상태에서는 “지금 고쳐야 할 버그”, “이미 끝난 작업”, “나중에 켤 기능”이 섞여 보인다.

따라서 앞으로는 아래처럼 구분해서 본다.

1. 현재 버그 수정 기준
2. 현재 문서 정리 기준
3. 미래 고도화 설계 기준

## 3. 지금 분리해서 봐야 할 작업 영역

### A. 지금 당장 고쳐야 할 감사 이슈

이 영역은 “기능 고도화와 무관하게 현재 운영 리스크를 줄이는 일”이다.

핵심 예시:

- `ad_team` 권한 격리
- `/resend` 권한 검증
- 로그인 세션 재생성
- `monitoring_time` 제출 정규화
- `match_score === 100` 강제 완화
- `approveRequest()` 상태 덮어쓰기
- 삭제/감사 로그 정합성
- 배포/백업 스크립트 문제

기준 문서:

- `./comprehensive-audit-roadmap-20260324.md`

### B. 지금 같이 정리해야 할 문서 이슈

이 영역은 “개발 방향을 다시 혼란스럽게 만들지 않기 위한 정리 작업”이다.

핵심 예시:

- `project-status.md`를 역사 문서로 재분류
- `db-schema.md` 최신 스키마 기준으로 재생성
- `improvement-plan.md`를 완료 기록/참고 문서로 명확화
- `HOW_TO_RUN.md`, `CLAUDE.md` 현행화

### C. 미래 고도화 기능

이 영역은 “지금 설계는 해두되, 현재 감사 버그 수정과 섞어 구현하지 않는 것”이다.

핵심 예시:

- 초 단위 광고 구간 자동 컷팅
- `clip_jobs`
- 관리자 ON/OFF
- `system_settings`
- `ffmpeg` 연계

기준 문서:

- `./clip-extraction-design.md`
- `./clip-extraction-implementation-plan.md`

## 4. 감사 결과를 지금 시점에서 어떻게 읽어야 하나

감사 결과는 아래처럼 해석한다.

### 4-1. 당장 수정 대상

`A-*`, `B-*`, `C-*` 중 현재 코드 결함과 직접 연결된 항목

특히 먼저 볼 것:

- `A-01` ~ `A-05`
- `A-06` ~ `A-10`
- `A-11` ~ `A-13`

### 4-2. 미래 설계를 반영해 해석해야 하는 항목

아래 항목은 광고 구간 컷팅 설계와 연결되므로, 단순 패치가 아니라 “미래 호환 수정”으로 본다.

- `A-05 monitoring_time 정규화`
- `A-06 match_score 100 강제`
- `A-07 100점 미만 결과 숨김`
- `A-09 삭제 정합성`
- `A-10 감사 로그 추적`
- `B-01 UTC/KST 저장 기준`

즉, 지금 고치더라도 나중에 다시 뒤엎지 않게 `HH:MM:SS`, feature flag, 추적 모델을 염두에 두고 수정해야 한다.

### 4-3. 지금 섞지 말아야 하는 것

아래는 같은 턴/같은 PR/같은 구현 묶음으로 합치지 않는 것이 좋다.

- P0 보안 수정 + 클립 기능 스키마 확장
- 권한 이슈 수정 + 관리자 시스템 설정 화면 추가
- `monitoring_time` 버그 수정 + `ffmpeg` 실행 기능 도입

이유:

- 검증 포인트가 완전히 다르다.
- 회귀가 섞여 원인 파악이 어려워진다.
- “현재 결함 수정”과 “미래 기능 추가”의 성공 기준이 다르다.

## 6. 앞으로 문서를 읽는 규칙

1. 버그 수정 전에는 먼저 `./comprehensive-audit-roadmap-20260324.md`를 본다.
2. 미래 고도화 작업 전에는 `./clip-extraction-design.md`와 `./clip-extraction-implementation-plan.md`를 본다.
3. `./project-status.md`는 현재 코드 기준 스냅샷으로 보고, 세부 우선순위 판단은 감사 문서에서 한다.
4. `./improvement-plan.md`는 현재 백로그가 아니라 “이미 한 작업 + 관련 설계 링크”로 본다.

## 7. 지금 기준 한 줄 정리

현재 가장 중요한 일은 “광고 구간 자동 컷팅 기능 구현”이 아니라, 감사 문서에 정리된 오픈 버그를 먼저 안정화하고, 문서 기준을 바로 세운 뒤, 그 다음에 feature flag를 깔고 미래 기능을 얹는 것이다.
